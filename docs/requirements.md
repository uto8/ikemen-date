# イケメンデート 要件定義書

最終更新: 2026-05-24

---

## 1. 概要・目的

「イケメンデート」は、イケメン男性と出会いたい女性ユーザー向けのWebマッチングアプリである。

- 男性はプロフィール画像・基本情報・イケメンタイプを登録し、女性に発見される
- 女性はギャラリー形式で男性一覧を閲覧し、気に入った相手にいいねを送る
- 双方がいいねを送り合うことでマッチングが成立し、テキストチャットが可能になる
- 課金・広告は一切なく、全機能を無料で提供する

---

## 2. ターゲットユーザー

### 女性ユーザー（主ターゲット）
- 18歳以上の女性
- イケメン男性と出会いたいと思っている
- iPhoneのSafariブラウザでサービスを利用する

### 男性ユーザー
- 18歳以上の男性
- 出会いを求めてプロフィールを登録する
- 女性からのいいねを受け取り、気に入った相手にいいねを返す

---

## 3. ユーザーストーリー

### 共通

- ユーザーとして、メールアドレス・パスワード・性別・生年月日を入力して新規登録したい。そうすることで、年齢確認と性別登録を一度に完了できる
- ユーザーとして、確認メール内のリンクをクリックしてメールアドレスを確認したい。そうすることで、アカウントを有効化できる
- ユーザーとして、登録済みのメールアドレスとパスワードでログインしたい。そうすることで、自分のアカウントにアクセスできる

### 女性ユーザー

- 女性として、男性ユーザーの一覧をギャラリー形式で閲覧したい。そうすることで、気になる相手を探せる
- 女性として、男性ユーザーの詳細プロフィールを確認したい。そうすることで、相手のことをよく知ってからいいねを送れる
- 女性として、気に入った男性にいいねを送りたい。そうすることで、マッチングのきっかけを作れる
- 女性として、自分へのいいね一覧を確認したい。そうすることで、自分に興味を持っている男性を把握できる
- 女性として、新しいいいねが届いた時にリアルタイムで通知バッジを受け取りたい。そうすることで、ページを更新しなくても新着いいねに気づける
- 女性として、マッチングした相手とテキストチャットをしたい。そうすることで、実際にコミュニケーションを取れる
- 女性として、マッチング相手から新着メッセージを受け取った時にリアルタイムで通知バッジを受け取りたい。そうすることで、ページを更新しなくても未読メッセージに気づける

### 男性ユーザー

- 男性として、プロフィール画像・基本情報・イケメンタイプを登録したい。そうすることで、女性ユーザーに自分を見つけてもらえる
- 男性として、女性ユーザーの一覧をギャラリー形式で閲覧したい。そうすることで、気になる相手を探せる
- 男性として、気に入った女性にいいねを送りたい。そうすることで、マッチングのきっかけを作れる
- 男性として、自分へのいいね一覧を確認したい。そうすることで、自分に興味を持っている女性を把握できる
- 男性として、新しいいいねが届いた時にリアルタイムで通知バッジを受け取りたい。そうすることで、ページを更新しなくても新着いいねに気づける
- 男性として、マッチングした相手とテキストチャットをしたい。そうすることで、実際にコミュニケーションを取れる
- 男性として、マッチング相手から新着メッセージを受け取った時にリアルタイムで通知バッジを受け取りたい。そうすることで、ページを更新しなくても未読メッセージに気づける

## 4. 機能要件

### 4.1 認証

#### 新規登録

年齢確認と性別登録を認証ステップで完結させる。サーバーサイドで検証してからSupabaseアカウントを作成することで、18歳未満のゴミアカウントが残留しない。

**登録フォーム入力項目**

| 項目 | 形式 | バリデーション |
|---|---|---|
| メールアドレス | メール形式 | 未登録であること |
| パスワード | テキスト | 英字と数字をそれぞれ1文字以上含む8文字以上 |
| 性別 | ラジオボタン（男性 / 女性） | いずれか1つ選択必須 |
| 生年月日 | 日付（YYYY-MM-DD） | 登録日時点で18歳以上であること |

**処理フロー**

1. フォーム送信 → Next.js Server Actionが全項目をサーバーサイドで検証する
2. 検証通過 → Supabase Admin APIでアカウントを作成する
3. アカウント作成成功 → Supabaseが確認メールを自動送信する
4. 「確認メールを送信しました」画面に遷移する（この時点ではまだログイン状態にしない）
5. ユーザーがメール内のリンクをクリック → メール確認完了 → オンボーディング画面に遷移する

**受け入れ条件**

```
Given: 未登録メールアドレス・有効なパスワード・性別・18歳以上の生年月日を全て入力した
When:  「登録する」ボタンを押した
Then:  サーバーサイドで検証後にアカウントが作成され、確認メール送信画面に遷移する
       Supabaseから入力したメールアドレスに確認リンク付きのメールが送信される

Given: すでに登録済みのメールアドレスを入力した
When:  「登録する」ボタンを押した
Then:  「このメールアドレスはすでに使用されています」とエラーメッセージが表示される
       アカウントは作成されない

Given: 生年月日が登録日時点で17歳以下になる値である
When:  「登録する」ボタンを押した
Then:  「18歳未満の方はご利用いただけません」とエラーメッセージが表示される
       アカウントは作成されない（サーバーサイドで拒否する）

Given: パスワードが7文字以下、または英字のみ・数字のみである
When:  「登録する」ボタンを押した
Then:  「パスワードは英字と数字をそれぞれ1文字以上含む8文字以上で入力してください」と表示される

Given: 性別を選択していない
When:  「登録する」ボタンを押した
Then:  「性別を選択してください」とエラーメッセージが表示される
```

#### メール確認

**受け入れ条件**

```
Given: 確認メール送信画面を表示している
When:  「確認メールを再送信する」ボタンを押した
Then:  同じメールアドレスに確認メールが再送信される
       再送信は60秒間に1回のみ可能とし、60秒以内の再押下はボタンを非活性にして残り秒数を表示する

Given: 確認メール内のリンクをクリックした（リンクが有効期限内の場合）
When:  リンク先ページが読み込まれた
Then:  メール確認が完了し、ログイン状態でオンボーディング画面に遷移する

Given: 確認メール内のリンクをクリックした（リンクの有効期限24時間が経過している場合）
When:  リンク先ページが読み込まれた
Then:  「確認リンクの有効期限が切れています。再度ログインして確認メールを再送信してください」と表示する

Given: メール確認が完了していないユーザーがログインを試みた
When:  「ログイン」ボタンを押した
Then:  「メールアドレスの確認が完了していません。確認メール送信画面へ」と表示し、確認メール送信画面に遷移する
```

#### ログイン

**受け入れ条件**

```
Given: メール確認済みのメールアドレスと正しいパスワードを入力した
When:  「ログイン」ボタンを押した
Then:  プロフィール登録が完了している場合はユーザー一覧画面に遷移する
       プロフィール登録が未完了の場合はオンボーディング画面に遷移する

Given: メールアドレスまたはパスワードが誤っている
When:  「ログイン」ボタンを押した
Then:  「メールアドレスまたはパスワードが正しくありません」とエラーメッセージが表示される
```

#### パスワードリセット

ログイン画面に「パスワードをお忘れの方はこちら」リンクを設ける。

**処理フロー**

1. リンクをタップ → メールアドレス入力画面に遷移する
2. メールアドレスを入力して送信 → Supabaseがパスワードリセットリンク付きメールを送信する
3. 「パスワードリセットメールを送信しました」画面を表示する
4. メール内のリンクをクリック（有効期限1時間）→ 新しいパスワード入力画面に遷移する
5. 新パスワードを入力して確定 → パスワードが更新され、ログイン画面に遷移する

**受け入れ条件**

```
Given: 登録済みのメールアドレスを入力して送信した
When:  送信が完了した
Then:  「パスワードリセットメールを送信しました」画面が表示される
       入力したメールアドレスにリセットリンク付きのメールが送信される

Given: 未登録のメールアドレスを入力して送信した
When:  送信が完了した
Then:  「パスワードリセットメールを送信しました」と表示する
       （登録済みかどうかをメールアドレス列挙攻撃から守るため、エラーと成功で表示を変えない）

Given: リセットリンクをクリックした（有効期限1時間以内）
When:  新パスワード入力画面が表示された
Then:  新しいパスワードを入力して確定するとパスワードが更新され、ログイン画面に遷移する
       新パスワードは登録時と同じバリデーション（英字と数字を各1文字以上含む8文字以上）を適用する

Given: リセットリンクをクリックした（有効期限1時間が経過している）
When:  リンク先ページが読み込まれた
Then:  「リンクの有効期限が切れています。再度パスワードリセットを行ってください」と表示する
```

### 4.2 プロフィール登録（オンボーディング）

メール確認完了後、プロフィール情報を登録するまでサービスの主要機能を利用できない。
性別と生年月日は新規登録フォームで取得済みのため、このステップでは入力しない。
オンボーディング画面はサインアップ時に登録した性別に応じて自動的に切り替わる。
入力項目は1画面1項目のステップ形式で表示する。画面上部に「現在ステップ番号 / 全ステップ数」をプログレスバーで示す。前のステップに戻る操作が可能で、入力済みの値は保持される。全ステップの入力完了後、最終ステップの「登録する」ボタンを押した時点で全項目を一括送信する。オンボーディング途中でブラウザを閉じた場合やページを離脱した場合、入力内容はすべて破棄される。再アクセス時はステップ1から再入力が必要になる。

#### 女性プロフィール登録

**必須項目**
| 項目 | 形式 | バリデーション |
|---|---|---|
| ニックネーム | テキスト | 1〜20文字。重複を許容する（一意性制約なし） |
| 居住地 | 都道府県（選択式・47都道府県） | 必須選択 |

**任意項目**
| 項目 | 形式 | バリデーション |
|---|---|---|
| プロフィール画像 | JPEG / PNG / WEBP | 10MB以下・1枚 |

**受け入れ条件**

```
Given: 全必須項目を入力した
When:  「登録する」ボタンを押した
Then:  プロフィールが保存され、ユーザー一覧画面に遷移する

Given: 必須項目のいずれかが未入力の状態である
When:  「登録する」ボタンを押した
Then:  未入力の項目に「必須項目です」とエラーメッセージが表示される

Given: 最終ステップで「登録する」ボタンを押したがサーバーエラーが発生した
When:  エラーレスポンスが返った
Then:  最終ステップの画面に留まり「登録に失敗しました。もう一度お試しください」とエラーメッセージが表示される
       テキスト項目の入力値は保持される
```

#### 男性プロフィール登録

**必須項目**
| 項目 | 形式 | バリデーション |
|---|---|---|
| プロフィール画像 | JPEG / PNG / WEBP | 10MB以下・1枚 |
| ニックネーム | テキスト | 1〜20文字。重複を許容する（一意性制約なし） |
| 居住地 | 都道府県（選択式・47都道府県） | 必須選択 |
| 職業 | テキスト | 1〜30文字 |
| 身長 | 数値（cm） | 100〜250の整数 |
| 自己紹介文 | テキスト（複数行） | 1〜300文字 |
| イケメンタイプ | チェックボックス（12種） | 1つ以上選択必須 |

**イケメンタイプ選択肢（12種）**
1. 王道アイドル系（例：平野紫耀）
2. 塩顔クール系（例：目黒蓮）
3. 犬系彼氏系（例：道枝駿佑）
4. ワイルド色気系（例：赤西仁）
5. 中性美容系（例：中島健人）
6. 韓国アイドル系（例：ラウール）
7. 年上お兄さん系（例：櫻井翔）
8. チャラモテ系（例：亀梨和也）
9. 爽やかスポーツ系（例：相葉雅紀）
10. 沼系ミステリアス系（例：二宮和也）
11. 陽キャムードメーカー系（例：菊池風磨）
12. かわいい系（例：岩橋玄樹）

**受け入れ条件**

```
Given: プロフィール画像を含む全必須項目を入力し、イケメンタイプを1つ以上選択した
When:  「登録する」ボタンを押した
Then:  プロフィールが保存され、ユーザー一覧画面に遷移する

Given: プロフィール画像を選択していない
When:  「登録する」ボタンを押した
Then:  「プロフィール画像は必須です」とエラーメッセージが表示される

Given: イケメンタイプをひとつも選択していない
When:  「登録する」ボタンを押した
Then:  「イケメンタイプを1つ以上選択してください」とエラーメッセージが表示される

Given: 最終ステップで「登録する」ボタンを押したがサーバーエラーが発生した
When:  エラーレスポンスが返った
Then:  最終ステップの画面に留まり「登録に失敗しました。もう一度お試しください」とエラーメッセージが表示される
       テキスト項目の入力値は保持される。プロフィール画像は再選択が必要になる
```

### 4.3 ユーザー一覧（ギャラリー）

- 自分と異性のユーザーを一覧表示する
- 表示形式：2カラムのカードグリッド
- 各カードに表示する情報：プロフィール画像・ニックネーム・年齢（閲覧日時点の日付を基準に生年月日から算出。誕生日当日から新しい年齢を表示する）・居住地
- 男性カードには加えてイケメンタイプのバッジを1つ表示する（複数選択時は12種類の定義順〈1〜12番〉で最も小さいインデックスのタイプを1つ表示する）
- 女性の場合、画像未設定ユーザーはデフォルトアバター画像を表示する
- 一覧はシステム登録順（新着順）で表示する

**受け入れ条件**

```
Given: プロフィール登録が完了しているユーザーがログインしている
When:  ユーザー一覧画面を開いた
Then:  自分と異性のユーザーが2カラムのカードグリッドで表示される

Given: ユーザー一覧画面を表示している
When:  任意のカードをタップした
Then:  そのユーザーの詳細プロフィール画面に遷移する
```

### 4.4 ユーザー詳細

- プロフィールの全項目を表示する
- 男性詳細に表示する項目：プロフィール画像・ニックネーム・年齢・居住地・職業・身長・自己紹介文・イケメンタイプ（選択済み全種）
- 女性詳細に表示する項目：プロフィール画像（またはデフォルトアバター）・ニックネーム・年齢・居住地
- 自分自身の詳細ページ（`/users/[自分のID]` を直接入力した場合）はプロフィールのみ表示し、いいねボタンを表示しない。ユーザー一覧には自分は表示されないため、通常この画面には到達しない
- プロフィールの編集機能は現バージョンでは実装しない（Section 8 項目2）。マイページ画面はプロフィール情報・ログアウト・退会への導線を持つ読み取り専用画面とする。プロフィール編集ボタンはどの画面にも設けない
- いいねボタンは以下の3状態を持つ

**いいねボタンの状態定義**

| 状態 | 条件 | 表示 | 操作可否 |
|---|---|---|---|
| 未送信 | 自分はいいねを送っておらず、相手からもいいねが来ていない | 「いいね」ボタン | 押下可 |
| 受信済み | 相手から自分へいいねが来ているが、自分はまだ送っていない | 「いいね」ボタン（未送信と同一表示） | 押下可（押すとマッチング成立） |
| 送信済み | 自分がすでにいいねを送った | 「いいね済み」ボタン | 押下不可（非活性） |

**受け入れ条件**

```
Given: 自分がいいねを送っておらず、相手からのいいねもない状態で詳細画面を開いた
When:  「いいね」ボタンを押した
Then:  いいねが送信され、ボタンが「いいね済み」の非活性状態に変わる

Given: 相手から自分へのいいねがある状態で詳細画面を開いた（自分はまだ送っていない）
When:  画面が表示された
Then:  「いいね」ボタンが活性状態で表示される（未送信状態と同一表示）

Given: 相手から自分へのいいねがある状態で「いいね」ボタンを押した
When:  押下が完了した
Then:  いいねが送信され、マッチングが成立する
       ボタンが「いいね済み」の非活性状態に変わる

Given: すでにいいねを送った相手の詳細画面を開いた
When:  画面が表示された
Then:  「いいね済み」の非活性ボタンが表示される

Given: 自分自身のユーザーIDで詳細画面URL（/users/[自分のID]）に直接アクセスした
When:  画面が表示された
Then:  プロフィール情報のみ表示され、いいねボタンは表示されない
```

### 4.5 いいね機能

- どちらの性別からも相手にいいねを送ることができる
- 上限なし・1人の相手に1回のみ送信可能
- いいねを取り消す機能はない

**受け入れ条件**

```
Given: ユーザーAがユーザーBにいいねを送った
When:  ユーザーBがユーザーAにいいねを送った
Then:  両者のマッチングが成立し、マッチング一覧にお互いが表示される

Given: ユーザーAがユーザーBにいいねを送った（Bはまだいいねしていない）
When:  マッチング状態を確認した
Then:  マッチングは成立していない
```

### 4.6 もらったいいね一覧

- 自分にいいねを送ってきたユーザーの一覧を表示する
- 表示形式：リスト形式（カード1列）
- 各カードに表示する情報：プロフィール画像・ニックネーム・年齢・居住地
- 新しくいいねを受け取った順（降順）で表示する

**受け入れ条件**

```
Given: 自分にいいねを送ってきたユーザーが存在する
When:  もらったいいね画面を開いた
Then:  いいねを送ってきたユーザーが一覧表示される

Given: まだ誰からもいいねをもらっていない
When:  もらったいいね画面を開いた
Then:  「まだいいねをもらっていません」と表示される

Given: もらったいいね一覧の任意のユーザーをタップした
When:  タップが完了した
Then:  そのユーザーの詳細プロフィール画面に遷移する

Given: BottomNavのいいねタブにバッジ（未読数）が表示されている
When:  もらったいいね一覧画面を開いた
Then:  バッジの未読カウントがゼロにリセットされ、バッジが非表示になる
```

### 4.7 マッチング一覧

- 双方がいいねを送り合ったユーザーの一覧を表示する
- 表示形式：リスト形式（カード1列）
- 各カードに表示する情報：プロフィール画像・ニックネーム・年齢・居住地
- 最終メッセージの送受信が新しい順（降順）で表示する。一度もメッセージを交わしていないマッチングはマッチング成立日時を基準に並べる
- 各カードに未読メッセージ件数バッジを表示する。未読が0件のカードにはバッジを表示しない

**受け入れ条件**

```
Given: マッチングが成立しているユーザーが存在する
When:  マッチング一覧画面を開いた
Then:  マッチング済みユーザーが一覧表示される

Given: まだマッチングしていない
When:  マッチング一覧画面を開いた
Then:  「まだマッチングしていません」と表示される

Given: マッチング一覧の任意のユーザーをタップした
When:  タップが完了した
Then:  そのユーザーとのチャット画面に遷移する

Given: 特定の相手から未読メッセージがある状態でマッチング一覧画面を開いた
When:  画面が表示された
Then:  未読メッセージがある相手のカードに未読件数バッジが表示される
       未読メッセージがない相手のカードにはバッジが表示されない
```

### 4.8 チャット

- マッチングが成立したユーザーとのみチャットができる
- 送受信できるのはテキストメッセージのみ
- 画像・スタンプ・ファイルの送信は不可
- 相手から受け取った未読メッセージは、チャット画面を開いた瞬間にまとめて既読にする（スクロール位置は問わない）
- プッシュ通知なし
- Realtimeの接続が切断された場合は画面上部に「再接続中...」バナーを表示し、自動再接続を試みる。再接続後は切断中に受信できなかったメッセージをサーバーから再取得し、バナーを非表示にする
- 1メッセージあたりの最大文字数：500文字（クライアント・サーバーサイド両方で検証する）
- メッセージは送信日時とともに表示する。当日のメッセージは「HH:MM」形式、前日以前は「M月D日 HH:MM」形式とする
- 自分のメッセージは右寄せ、相手のメッセージは左寄せで表示する

**受け入れ条件**

```
Given: マッチング済みの相手とのチャット画面を開いている
When:  テキストを入力して「送信」ボタンを押した
Then:  送信したメッセージが画面右側にリアルタイムで表示される

Given: チャット画面を開いている
When:  相手がメッセージを送信した
Then:  相手のメッセージが画面左側にリアルタイムで表示される（ページリロード不要）

Given: 入力欄が空（スペースのみを含む場合も含む）の状態である
When:  「送信」ボタンを押した
Then:  メッセージは送信されない。「送信」ボタンは入力欄が空の場合は非活性状態にする

Given: 501文字以上のテキストを入力している
When:  入力欄に文字を入力した
Then:  501文字目以降は入力できない（入力欄でそれ以上の文字を受け付けない）
       入力欄の下部に「500文字以内で入力してください（現在N文字）」と残り文字数を表示する

Given: マッチングしていない相手のURLに直接アクセスした
When:  チャット画面が表示されようとした
Then:  アクセスが拒否され、マッチング一覧画面にリダイレクトされる

Given: 相手から未読メッセージがある状態でチャット画面を開いた
When:  チャット画面が表示された
Then:  そのマッチングで相手から受け取った全未読メッセージが既読になる
       BottomNavのマッチングタブのバッジカウントが既読にした件数分減少する

Given: 同一アカウントが複数のデバイスで同じチャット画面を開いた
When:  チャット画面が開いた
Then:  全デバイスで既読処理が実行される
       is_read = true へのUPDATEは冪等なため、複数デバイスからの並行実行による不整合は生じない
```

### 4.9 退会

退会処理ではプロフィールといいねを即時削除する。マッチングとチャット履歴は削除せず保持することで、相手ユーザーが過去の会話を参照できる状態を維持する。

**退会時のデータ処理**

| 対象 | 処理 | 詳細 |
|---|---|---|
| プロフィール | 即時削除 | profiles テーブルのレコード・Supabase Storage の画像ファイル |
| いいね | 即時削除 | 送受信した全 likes レコード |
| マッチング | 保持（参照をNULL化） | matches.user1_id / user2_id の該当列を NULL に設定。相手のマッチング一覧には表示されなくなる。PostgreSQLはNULLと値の比較結果をNULL（unknown）として扱うため、NULL化後もCHECK制約 `user1_id < user2_id` には違反しない |
| チャット履歴 | 保持（参照をNULL化） | messages.sender_id の該当列を NULL に設定。退会済みユーザーが送ったメッセージはニックネームを「退会済みユーザー」、アバターをデフォルトアバター画像で表示する |

**補足：部分的な失敗の扱い**

Supabase Storageの画像ファイル削除が失敗した場合でも、DBのprofilesレコード削除はそのまま完了扱いとする。孤立した画像ファイルはFreeプランのストレージ容量内で許容する。

**受け入れ条件**

```
Given: ログイン済みのユーザーが退会操作を行った
When:  確認ダイアログで「退会する」を選択した
Then:  プロフィールといいねが即時削除され、退会者はトップページにリダイレクトされる
       マッチング・チャット履歴は保持され、相手側の参照列がNULLに設定される

Given: ユーザーAが退会した
When:  ユーザーBがマッチング一覧を開いた
Then:  ユーザーAとのマッチングは一覧に表示されない
       （matches の user_id が NULL のレコードはマッチング一覧のクエリから除外する）

Given: ユーザーAが退会した時点でユーザーBがユーザーAとのチャット画面を開いていた
When:  ユーザーAの退会処理が完了した
Then:  ユーザーBのチャット画面に「相手が退会しました。このチャットは利用できません」のバナーを表示する
       過去のメッセージは引き続き表示する（ユーザーAが送ったメッセージは「退会済みユーザー」として表示する）
       テキスト入力欄と送信ボタンを非活性にする

Given: ユーザーAが退会した後にユーザーBがチャット画面URLに直接アクセスした
When:  チャット画面が表示された
Then:  過去のメッセージを表示する（ユーザーAが送ったメッセージは「退会済みユーザー」として表示する）
       「相手が退会しました。このチャットは利用できません」のバナーを表示する
       テキスト入力欄と送信ボタンを非活性にする

Given: 退会済みユーザーのメールアドレスで新規登録を試みた
When:  「登録する」ボタンを押した
Then:  通常通り新規登録が完了する（退会済みアドレスは再利用可能）
```

### 4.10 通知バッジ（BottomNav）

BottomNavの「いいね」タブと「マッチング」タブに未読件数のバッジを表示する。バッジはSupabase Realtimeにより、ページリロードなしでリアルタイムに更新される。

**バッジ仕様**

| タブ | カウント対象 | クリアタイミング |
|---|---|---|
| いいね | 未読のいいね受信数 | もらったいいね一覧画面（S12）を開いた瞬間（全件一括クリア） |
| マッチング | 全マッチングの未読メッセージ件数（自分の送信・退会済み相手のメッセージを除く） | 各チャット画面（S14）を開いた瞬間（そのマッチング分のみクリア） |

- バッジは1〜99件のときはそのままの件数を表示し、100件以上のときは「99+」と表示する
- いいね一覧（S12）表示中はいいねタブのバッジを非表示にする
- マッチング一覧（S13）表示中はマッチングタブのバッジを非表示にするが、クリアはしない。クリアは各チャット画面（S14）を開いた時のみ行う
- バッジの未読カウントはクライアントがDBにCOUNTクエリを発行して取得する。Supabase Realtimeでlikesおよびmessagesテーブルへの変更通知を受け取った際に再クエリを実行し、表示をリアルタイムに更新する

**受け入れ条件**

```
Given: 別のユーザーが自分にいいねを送信した
When:  自分がもらったいいね一覧画面以外の任意の画面を表示している
Then:  BottomNavのいいねタブに未読いいね数のバッジがリアルタイムで表示される
       バッジの数字は未読のいいね受信件数と一致する

Given: 未読のいいねが100件以上ある状態である
When:  BottomNavのいいねタブのバッジを確認した
Then:  バッジに「99+」と表示される

Given: ログイン後に一度もいいね一覧を開いておらず、過去にいいねを受け取っていた
When:  いいね一覧以外の任意の画面を表示した
Then:  累積したいいね受信数の合計がバッジに表示される

Given: BottomNavのいいねタブにバッジが表示されている
When:  もらったいいね一覧画面（S12）を開いた
Then:  バッジの未読カウントがゼロにリセットされ、バッジが非表示になる

Given: もらったいいね一覧画面（S12）を表示している
When:  BottomNavのいいねタブを確認した
Then:  バッジは表示されない（現在表示中の画面が対象タブと同一のため）

Given: マッチング相手が新しいメッセージを送信した
When:  自分がそのチャット画面（S14）以外の任意の画面を表示している
Then:  BottomNavのマッチングタブのバッジカウントが送信されたメッセージ件数分リアルタイムで増加する

Given: 退会済みユーザーとのチャットに未読メッセージが存在する
When:  BottomNavのマッチングタブのバッジを確認した
Then:  退会済みユーザーからのメッセージは未読カウントに含まれない

Given: 未読メッセージが100件以上ある状態である
When:  BottomNavのマッチングタブのバッジを確認した
Then:  バッジに「99+」と表示される

Given: BottomNavのマッチングタブにバッジが表示されている
When:  特定のチャット画面（S14）を開いた
Then:  そのマッチングで相手から受け取った全未読メッセージが既読になる
       BottomNavのマッチングタブのバッジカウントが既読にした件数分減少する
       他のマッチングに未読メッセージが残っている場合はバッジが継続して表示される

Given: マッチング一覧画面（S13）を表示している
When:  BottomNavのマッチングタブを確認した
Then:  バッジは表示されない（現在表示中の画面が対象タブと同一のため）
       S13を開いてもバッジはクリアされない。クリアは各チャット画面（S14）を開いた時のみ行う
```

## 5. 非機能要件

### 5.1 規模・パフォーマンス

| 項目 | 要件 |
|---|---|
| 想定ユーザー数 | 最大100人 |
| 最大同時接続数 | 100接続 |
| 画像アップロード上限 | 1ファイルあたり10MB以下 |
| チャットのリアルタイム性 | Supabase Realtimeによるリアルタイム配信（ポーリングは使用しない） |

### 5.2 対応環境

| 項目 | 要件 |
|---|---|
| 優先ブラウザ | Safari（iOS・最新版） |
| 追加対応ブラウザ | Chrome（Android・最新版）、Chrome（PC・最新版） |
| レスポンシブ対応 | 必須。モバイルファーストで設計し、PCでも正常に閲覧できること |
| 最小画面幅 | 375px（iPhone SE相当） |

### 5.3 セキュリティ

| 項目 | 要件 |
|---|---|
| 認証方式 | メールアドレス＋パスワード（Supabase Auth） |
| パスワード要件 | 英字と数字をそれぞれ1文字以上含む8文字以上 |
| メール確認 | 新規登録後にSupabaseが確認メールを送信する。確認完了前はログイン状態にしない |
| 年齢確認 | 新規登録フォームで生年月日を収集し、Next.js Server Actionがアカウント作成前にサーバーサイドで18歳未満を拒否する。フロントエンドのバリデーションのみには依存しない |
| 性別の保持 | 性別は新規登録時にDBに保存し、オンボーディング・一覧フィルタリング・RLSポリシーで参照する |
| データアクセス制御 | Row Level Security（RLS）により、自分以外のデータを書き換え・削除できない |
| チャット閲覧制限 | マッチングしていないユーザーのチャット履歴は取得できない（RLSで制御） |
| 画像の公開範囲 | プロフィール画像はSupabase StorageのPublicバケットに保存する。画像URLを知っていれば非ログイン状態でも閲覧可能になるが、マッチングアプリとして顔写真を公開する性質上、許容リスクとして扱う |
| 通信 | 全通信をHTTPS経由で行う（Vercelデフォルト） |

### 5.4 インフラ・デプロイ

| 項目 | 要件 |
|---|---|
| フロントエンド | Next.js 16（App Router） |
| バックエンド・DB・Auth | Supabase（Freeプラン） |
| ファイルストレージ | Supabase Storage（Publicバケット。プロフィール画像の保存） |
| Realtimeの制約 | Supabase Freeプランの同時Realtime接続上限は200。想定同時接続数100に対して余裕はあるが、1ユーザーが複数タブを開いた場合は接続数を複数消費する点に注意する |
| デプロイ先 | Vercel |
| 環境 | 本番環境のみ（ステージング環境なし） |

## 6. スコープ外（やらないこと）

以下の機能は本プロジェクトでは実装しない。

| 項目 | 理由 |
|---|---|
| 課金・サブスクリプション | 完全無料サービスのため |
| SNSログイン（Google / LINE / Apple） | 認証はメール＋パスワードのみ |
| SMS認証・本人確認書類による年齢確認 | 自己申告（生年月日）のみ |
| プッシュ通知 | 実装対象外 |
| 画像・スタンプ・ファイルのメッセージ送信 | テキストのみ |
| ブロック・通報機能 | 実装対象外 |
| 管理者画面 | 実装対象外 |
| プロフィール画像の審査 | 審査なし（アップロードのみ） |
| 検索・フィルタリング機能 | 実装対象外 |
| マッチング相手との動画・音声通話 | 実装対象外 |
| いいねの取り消し | 実装対象外 |
| LGBT・同性マッチング | 異性間マッチングのみ |
| ステージング環境 | 本番環境のみ |

## 7. 用語集

| 用語 | 定義 |
|---|---|
| 性別 | 新規登録フォームで選択する「男性」または「女性」の区分。一覧表示・RLSポリシー・オンボーディングフォームの分岐に使用する |
| いいね | 相手ユーザーへの好意を示すアクション。1人の相手に1回のみ送信可能 |
| マッチング | 2人のユーザーが互いにいいねを送り合った状態。マッチング成立後にチャットが可能になる |
| オンボーディング | 新規登録直後に行うプロフィール登録フローのこと |
| イケメンタイプ | 男性ユーザーが自身の外見イメージを12種類から選択するチェックボックス項目 |
| ギャラリー | ユーザーを2カラムのカードグリッドで表示する一覧画面の形式 |
| デフォルトアバター | プロフィール画像を未設定のユーザーに表示するシステム標準画像 |
| RLS | Row Level Security。Supabaseのデータベース行レベルアクセス制御機能 |
| 既読 | 受信者がチャット画面を開いた時点で「読んだ」と判定された状態。`messages.is_read = true` で管理する |
| 未読メッセージ | `messages.is_read = false` かつ送信者が自分以外のメッセージ。BottomNavのマッチングタブバッジのカウント対象になる |
| 通知バッジ | BottomNavのいいねタブ・マッチングタブに表示される未読件数の数字付き丸表示。0件の場合は非表示 |
